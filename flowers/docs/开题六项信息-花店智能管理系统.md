# 基于微信小程序的花店智能管理系统——开题六项信息

## 1. 选题目的、意义

本课题面向中小型花店在日常经营中的真实痛点：一是线下接单、库存登记和价格核算依赖人工，效率较低且容易出错；二是鲜花具有易损耗、保鲜周期短的特点，传统经验式进货容易导致旺季缺货与淡季积压并存；三是用户消费场景逐步迁移到移动端，花店需要建设低成本、可持续迭代的线上经营能力。

本项目以“微信小程序 + Spring Boot + MySQL + Python 分析服务”为技术主线，构建覆盖“商品展示—下单支付—库存管理—数据分析—补货决策”的业务闭环。通过系统建设，能够实现订单流程标准化、库存过程可追踪、销量趋势可预测，提升花店运营效率和客户体验。

课题意义主要体现在实践意义、技术意义和应用意义三个维度：在实践层面，本系统为中小花店提供可落地、可复用的数字化方案，能够降低人工管理成本并减少运营风险；在技术层面，验证了 Java 业务服务与 Python 数据分析协同架构在小微商业场景中的可行性与扩展性；在应用层面，围绕鲜花行业“易腐、节日波动明显”的经营特点，引入 FEFO（先到期先出）和预测补货机制，有助于提升花店行业的信息化与精细化管理水平。

## 2. 研究内容

1. 完成花店业务场景需求分析，明确用户端、商家端、管理员端的功能边界与流程关系。
2. 设计系统总体架构，形成“前端交互层—后端业务层—数据存储层—分析决策层”四层架构。
3. 设计并实现核心功能模块：
   - 用户模块：商品浏览、购物车、下单、订单查询、评价反馈；
   - 商家模块：商品管理、库存批次管理、订单处理、库存预警；
   - 管理模块：用户与商家管理、角色权限、参数配置；
   - 智能模块：销量预测、补货建议、商品推荐。
4. 建立数据库与接口规范，保障订单、库存、支付状态等核心数据的一致性和可追溯性。
5. 开展系统测试与性能验证，对核心链路（下单→支付→库存扣减/回滚）进行完整性与稳定性评估。

## 3. 主要任务

1. **文献与需求阶段**：收集花店管理、微信小程序、电商库存管理与预测算法相关资料，形成需求说明。
2. **系统设计阶段**：完成用例图、功能结构图、数据库 E-R 设计、接口清单与技术路线设计。
3. **系统实现阶段**：
   - 实现 Spring Boot 后端 REST API；
   - 实现微信小程序与 Web 管理端主要页面；
   - 实现 MySQL 数据库建表、种子数据与核心事务逻辑；
   - 实现 Python 销量预测与补货建议服务。
4. **联调测试阶段**：完成前后端联调、典型业务压测、异常场景处理（取消订单、退款回滚、库存释放）。
5. **论文与答辩阶段**：完成毕业论文撰写、系统演示材料整理、答辩 PPT 与测试证据准备。

## 4. 资料、数据、技术等方面的要求

（1）资料方面：本课题需系统查阅花店管理、电商系统设计、库存控制、时间序列预测及微信小程序开发等相关资料，参考文献以中英文结合为原则，总量控制在 17 篇左右，其中近五年文献占主要比例；同时在系统设计与实现过程中，需重点参考微信小程序、Spring Boot、MySQL、Pandas/NumPy 等官方技术文档，确保技术路线规范、实现过程可复现。

（2）数据方面：系统需要构建完整的业务数据体系，覆盖商品、花材、BOM、订单、订单明细、库存批次、库存流水、用户评价等核心数据对象，并通过真实样例数据与模拟造数脚本结合的方式形成可用于功能验证与结果分析的数据集；数据应具备可追踪性，能够清晰反映“下单—支付—库存扣减/回滚”的全过程变化。

（3）技术方面：项目采用“微信小程序前端 + Spring Boot 后端 + MySQL 数据库 + Python 分析服务”的分层架构，要求后端能够稳定提供 REST API，前端完成主要业务页面交互，数据库保证事务一致性与查询性能，分析层可输出销量预测与补货建议结果；同时需提供接口文档、部署说明与测试记录，保证系统达到可运行、可演示、可答辩的技术水平。

## 6. 完成形式（论文、作品、软件等）

本课题最终成果采用“论文 + 软件系统 + 配套文档”形式提交：

1. **毕业论文**：完成《基于微信小程序的花店智能管理系统设计与实现》正文及附录；
2. **可运行软件系统**：包含微信小程序用户端、Web 管理端、Spring Boot 后端、Python 分析服务；
3. **工程源码与数据库脚本**：提交完整源代码、建表脚本、初始化数据与关键配置说明；
4. **技术文档**：包含需求说明、系统设计说明、接口文档、测试报告；
5. **答辩材料**：系统演示视频（或现场演示流程）与答辩 PPT。
