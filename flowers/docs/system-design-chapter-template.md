## 3 系统设计

### 3.1 总体设计目标

本系统以鲜花行业高损耗、强时效、节日波动大等特点为背景，构建“交易-库存-分析-决策”闭环。系统设计目标如下：

1. 提升订单处理效率，保障下单至履约流程可追踪。
2. 保证库存数据一致性，降低超卖与错卖风险。
3. 基于历史数据输出销量预测与补货建议，降低库存浪费。
4. 在微信生态内提供低成本、可扩展、可运维的数字化方案。

### 3.2 系统架构设计

系统采用分层架构：展示层、业务层、数据层、分析层。

- 展示层：微信小程序和 Vue 3 管理后台。
- 业务层：Spring Boot 提供 REST API，承载订单、库存、商品等核心服务。
- 数据层：MySQL 存储主业务数据，保证事务一致性。
- 分析层：Python 服务负责销量预测与推荐计算，并回写分析结果。

### 3.3 功能模块设计

#### 3.3.1 用户端模块

实现商品浏览、分类检索、购物车、下单支付、订单追踪、评价反馈等功能。

#### 3.3.2 商家端模块

实现商品与花材维护、BOM 管理、订单处理、库存预警、销售统计。

#### 3.3.3 管理员端模块

实现用户与商家管理、角色权限、日志审计与参数配置。

### 3.4 数据库设计

#### 3.4.1 核心实体

系统核心实体包括：用户、花材、商品、BOM、库存批次、订单、订单明细、库存锁、预测结果、补货建议、推荐结果。

#### 3.4.2 关键关系

1. 商品与花材通过 BOM 形成多对多关系。
2. 订单与订单明细是一对多关系。
3. 订单与库存锁流水是一对多关系。
4. 花材与库存批次是一对多关系。

### 3.5 库存一致性流程设计

为保证库存精度，系统采用“锁定-确认-回滚”机制：

1. 下单创建时，按 FEFO 规则锁定批次库存。
2. 支付成功后，将锁定库存转为正式扣减。
3. 未支付取消订单释放锁定库存。
4. 已支付退款订单执行库存回补。

### 3.6 智能预测与补货策略设计

系统采用 Prophet 与移动平均的组合策略。若训练数据充分，使用 Prophet 建模；若样本不足，则退化为移动平均。补货策略采用安全库存与再订货点模型：

`ROP = Davg * L + SS`

`SS = Z * sigma * sqrt(L)`

并给出建议补货量：

`Q = max(0, 未来需求 + SS - 当前库存 - 在途量)`

### 3.7 推荐模块设计

推荐模块采用“规则 + 统计”混合策略，融合用户历史购买偏好、近期热销程度及库存可用性，输出可解释的推荐理由，提升转化率并辅助去库存。
